# SmartReceipt
# מערכת Serverless חכמה לניהול וניתוח קבלות בענן

## 📝 תיאור הפרויקט
פרויקט גמר במסגרת קורס **ניהול מערכות ענן (AWS)**, המכללה האקדמית להנדסה עזריאלי.
מערכת זו מבוססת על ארכיטקטורת Serverless מלאה ב-AWS ומאפשרת למשתמשים להעלות תמונות של קבלות פיזיות דרך ממשק דפדפן סטטי ומעוצב. המערכת קולטת את התמונה, מריצה שכבת וולידציה והגנה מפני קבצי ספאם, מנתחת את הקבלה באמצעות מודל בינה מלאכותית (OCR) לחילוץ נתונים פיננסיים, ומקטלגת את התוצאות באופן אוטומטי בבסיס נתונים לצורך מעקב וניהול הוצאות בזמן אמת.

## 🗺️ ארכיטקטורת המערכת וזרימת הנתונים (Data Flow)
המערכת פועלת בצורה חסכונית ומאובטחת לפי השלבים הבאים:
1. **ממשק משתמש (Frontend):** דף אינטרנט יחיד (Single Page Application) המבוסס על HTML5 ,CSS3 ו-JavaScript אסינכרוני, המאוחסן ומוגש באמצעות **Amazon S3 Static Website Hosting**. הממשק מעוצב ותומך בהצגת נתונים פיננסיים רב-מטבעיים (עם דגש על דולר ארה"ב).
2. **שער ה-API:** הממשק ממיר את התמונה לפורמט קריא (Base64) ושולח בקשת HTTP POST אל **Amazon API Gateway** (בתצורת HTTP API) המנהל את נתיב הגישה המאובטח `/upload` תחת הגדרות CORS.
3. **עיבוד ולוגיקה (Backend):** פונקציית **AWS Lambda** (סביבת Python 3.12) מתעוררת מהאירוע ומפענחת את התמונה מהזיכרון ישירות ל-Bytes.
4. **ראייה ממוחשבת (AI & OCR):** ה-Lambda מעבירה את קובץ ה-Bytes לניתוח בשירות **Amazon Rekognition** שמחלץ את שורות הטקסט הגולמיות.
5. **שער וולידציה וסיווג חכם (Validation & Classification Gate):** קוד הפונקציה מריץ אלגוריתם חכם המאמת נוכחות של מילות מפתח פיננסיות וסימני מטבע בינלאומיים (למשל: סה"כ, מע"מ, קבלה, ש"ח, וכן $, USD, Dollar). 
   - *במידה והקובץ אינו קבלה:* המערכת עוצרת מייד ומחזירה שגיאה 400 ללקוח מבלי לשמור דבר – ובכך חוסכת עלויות אחסון ומשאבים.
   - *במידה והקובץ תקין:* מבוצע חילוץ נתונים ממוקד (שם עסק, סכום, מע"מ, אמצעי תשלום). לצד זאת, מופעל **מנוע סיווג אוטומטי (Classification Engine)** המנתח את תוכן הטקסט אל מול מילון מונחים מוגדר מראש, ומקטלג את הקבלה לאחת מחמש קטגוריות ניהוליות והוצאות משרדיות באנגלית (`Marketing and Advertising`, `Food and Hospitality`, `Travel and Transportation`, `IT and Equipment`, `Maintenance and General Expenses`).
6. **אחסון וגיבוי קבצים:** התמונה המקורית נשמרת תחת מזהה ייחודי (UUID) בתוך באקט **Amazon S3 Storage** מאובטח וחסום לגישה ציבורית.
7. **בסיס הנתונים:** המטא-דאטה שחולץ (כולל הקטגוריה החדשה שסווגה) נשמר בשורה ייחודית בטבלת **Amazon Amazon DynamoDB** (NoSQL) המאפשרת שליפה וניהול מהירים.

## 🛠️ שירותי AWS בהם נעשה שימוש
* **Amazon S3:** משמש בשני רכיבים נפרדים - אחד לאירוח האתר הסטטי (ציבורי) ואחד לאחסון מאובטח של הקבלות (פרטי).
* **Amazon API Gateway:** ניהול ה-Endpoints ואבטחת נתיב התקשורת.
* **AWS Lambda (Python 3.12):** הרצת קוד ה-Backend ללא שרתים וניהול זרימת הנתונים והלוגיקה הפיננסית.
* **Amazon DynamoDB:** מסד נתונים מהיר וגמיש לשמירת רשומות הקבלות (Partition Key: `ReceiptId`).
* **Amazon Rekognition:** שירות הבחירה המנוהל (AI/ML) המשמש לביצוע ה-OCR וחילוץ הטקסט מהתמונה.
* **AWS IAM:** ניהול הרשאות קפדני לפי עקרון ההרשאה המינימלית (Least Privilege) המאפשר ל-Lambda גישה מוגדרת למשאבים הרלוונטיים בלבד.
* **Amazon CloudWatch:** ניטור זמני הריצה, מעקב לוגים וביצוע אופטימיזציית משאבים.

## 🚀 הוראות התקנה ופריסה (Deployment Instructions)
1. **הקמת בסיס הנתונים:** יצירת טבלת DynamoDB בשם `Receipts` עם Partition Key מסוג String בשם `ReceiptId`.
2. **הקמת רכיבי האחסון:** 
   - יצירת S3 Bucket פרטי לאחסון קבלות (לעדכן את השם `smartreceipt-uploads-final-2026` בקוד ה-Lambda).
   - יצירת S3 Bucket שני, הפעלת אפשרות ה-Static Website Hosting והעלאת קובץ ה-`index.html`.
3. **יצירת פונקציית ה-Lambda:**
   - הקמת פונקציה חדשה ב-AWS Lambda מבוססת Python 3.12.
   - הדבקת קוד ה-Backend מתוך הקובץ `lambda_function.py`.
   - הגדרת ה-Execution Role ב-IAM עם הרשאות ל-DynamoDB, S3, Rekognition ו-CloudWatch.
   - תחת לשונית Configuration, העלאת ה-Timeout ל-30 שניות והרחבת הזיכרון ל-512MB.
4. **חשיפת ה-API:** הקמת HTTP API ב-API Gateway, יצירת נתיב `POST /upload`, חיבורו לפונקציית הלמדא, והפעלת הגדרות CORS. יש להעתיק את ה-URL שנוצר ולהדביק אותו תחת המשתנה **`API_GATEWAY_URL`** בתוך בלוק ה-`script` בקובץ ה-`index.html`.

## 📊 אתגרים ארכיטקטוניים ופתרונם
* **מגבלות אזוריות (Region Availability):** בתחילה ניסינו לפרוס את שירותי ה-AI באזור שטוקהולם (`eu-north-1`), אך נתקלנו בשגיאות חיבור מאחר ושירות Amazon Rekognition לא היה זמין שם באופן מלא. הבעיה נפתרה על ידי העברת כל תשתית הפרויקט לאזור צפון וירג'יניה (`us-east-1`).
* **פסק זמן בעיבוד (Lambda Timeout):** ברירת המחדל של Lambda (כ-3 שניות) גרמה לחיתוך הריצה בעת עיבוד ופענוח תמונות קבלות כבדות וברזולוציה גבוהה. הבעיה נפתרה על ידי הגדלת ה-Timeout ל-30 שניות דרך ה-Console והרחבת הזיכרון ל-512MB, דבר ששיפר משמעותית גם את מהירות העיבוד של מודל ה-AI.
* **התמודדות עם העלאת תמונה שאינה קבלה (Input Validation):** אחד האתגרים המרכזיים היה מניעת עיבוד שגוי של תמונות שאינן קבלות. הפתרון מומש בפונקציית ה-Lambda באמצעות מנגנון וולידציה שפועל מיד לאחר חילוץ הטקסט על ידי Rekognition - אם לא זוהו לפחות 3 שורות טקסט ומילת מפתח פיננסית אחת לפחות (כגון 'סך', 'מע״מ', 'total'), המערכת מחזירה שגיאת 400 ועוצרת את התהליך לחלוטין, מבלי לכתוב דבר ל-S3 או ל-DynamoDB."
